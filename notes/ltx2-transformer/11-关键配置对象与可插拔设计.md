# 11 关键配置对象与可插拔设计

## 这个机制解决什么问题

前面几篇主要在看“模型怎么跑”。这一篇换一个角度：为什么 LTX-2 可以在不改 block 主结构的前提下，切不同的 attention backend、不同的编译策略，甚至不同的预处理行为。

答案就在几组配置对象和可插拔接口里。

## 它在整体调用链中的位置

这些配置对象并不直接产生输出张量，但它们决定了：

- block 初始化时装入什么结构参数；
- `Attention` 内部接上什么 callable；
- `torch.compile` 相关能力怎样包到现有模型上。

所以它们属于“结构与执行策略的连接层”。

重点阅读文件：

- `packages/ltx-core/src/ltx_core/model/transformer/transformer.py`
- `packages/ltx-core/src/ltx_core/model/transformer/ops.py`
- `packages/ltx-core/src/ltx_core/model/transformer/attention.py`
- `packages/ltx-core/src/ltx_core/model/transformer/compiling.py`

## 关键类与函数

- `TransformerConfig`
- `TransformerOpsConfig`
- `DEFAULT_TRANSFORMER_OPS`
- `AttentionOps`
- `CompilationConfig`
- `compile_transformer()`
- `_SeqDynamicMarkingProcessor`

## 数据流或张量流

这一篇虽然偏配置，但仍然可以看成一条“参数如何流向模块”的路径：

`LTXModel.__init__ 参数 -> TransformerConfig / TransformerOpsConfig -> BasicAVTransformerBlock / Attention -> 真实运行行为`

编译路径则是：

`CompilationConfig -> compile_transformer() -> 替换 transformer_blocks + 包装 forward`

也就是说，这些配置对象本身不做注意力计算，但它们决定了“谁来算、以什么方式算、运行前还要包哪些运行时逻辑”。

## 关键源码分段解析

### 1. `TransformerConfig` 负责描述一条流的结构参数

这个 dataclass 比较像“block 层面的结构合同”，里面会放：

- `dim`
- `heads`
- `d_head`
- `context_dim`
- `apply_gated_attention`
- `cross_attention_adaln`

它的作用不是替代模型总配置，而是把“video 流需要什么参数”“audio 流需要什么参数”明确打包后，再交给 `BasicAVTransformerBlock` 初始化。

### 2. `TransformerOpsConfig` 负责描述一层应接入哪些实现

与 `TransformerConfig` 相比，`TransformerOpsConfig` 更偏执行策略：

- `attention_ops`
- `ada_zero_function`
- `post_sa_function`

尤其 `from_functions()` 很重要，因为它允许调用方用枚举或 callable 局部覆盖实现，而不必手工拼一整套完整对象。

### 3. `AttentionOps` 是 Attention 层自己的可插拔入口

这一层再往下，会把 attention 需要的几个关键 callable 固定下来：

- unmasked attention
- masked attention
- preattention
- gated attention

这就形成了一个很清楚的分层：

- `TransformerOpsConfig` 面向 block；
- `AttentionOps` 面向具体 attention 模块；
- 更底层的 backend callable 面向 kernel 选择。

### 4. `CompilationConfig` 和 `compile_transformer()` 说明“编译”也是附加层，不是结构本体

`compile_transformer()` 并没有重写模型结构，而是：

1. 把 `transformer_blocks` 逐个包上 `torch.compile`；
2. 用 `_SeqDynamicMarkingProcessor` 包住原 `block_input_processor`；
3. 再给 `forward` 套一层 patched 包装。

这说明编译优化在 LTX-2 里被设计成“外挂式增强”，而不是硬编码进 block 本体。

### 5. `_SeqDynamicMarkingProcessor` 是一个很有代表性的工程细节

它不是模型数学的一部分，却很能体现工程成熟度：为了让每层 block 的 compile artifact 更具 shape polymorphism，代码会在进入编译边界前先标记动态维度。

这类逻辑如果直接塞进 block 里，会严重污染核心阅读体验；单独包成 processor，则既保留了能力，又保持了主路径整洁。

## 设计动机与实现取舍

这一层最值得理解的不是“某一个配置项的默认值”，而是 LTX-2 把“模型结构”和“底层算子实现”拆开的设计思路。这样同一个 block 结构，才能在不同 attention backend 或 compile 策略之间切换。

这种拆法的好处非常明显：

- 想换 attention backend，不必重写 block；
- 想试编译优化，不必改 preprocessor 与 attention 结构；
- 单流和双流共享结构时，也能保留不同执行策略。

代价则是抽象层级更多：

- 初学者需要分清“结构参数”和“执行参数”；
- 需要接受某些行为不在 block 里写死，而是从配置中注入。

但从长期维护角度看，这几乎是必要代价。因为 LTX-2 面对的不是一个固定环境，而是多种 GPU、可选加速包、可选 compile 策略。
