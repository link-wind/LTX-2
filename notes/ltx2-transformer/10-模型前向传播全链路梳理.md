# 10 模型前向传播全链路梳理

## 这个机制解决什么问题

到这一篇为止，预处理、AdaLN、RoPE、Attention、FFN、单层 block、双流交互都已经拆开看过了。现在要做的不是再学一个新机制，而是把这些零件重新接回一次真实的 `LTXModel.forward()`。

本篇的目标是让你回到总模型时不再迷路。

## 它在整体调用链中的位置

它覆盖的是从 `LTXModel.forward()` 入口到输出张量返回的完整主线。

可以把完整路径记成：

`Modality -> 预处理器 -> TransformerArgs -> 多层 BasicAVTransformerBlock -> norm/proj_out -> 最终输出`

如果是双流模型，这条路径会并行跑 video 和 audio 两条分支，并在 block 内中途发生交互。

重点阅读文件：

- `packages/ltx-core/src/ltx_core/model/transformer/model.py`
- `packages/ltx-core/src/ltx_core/model/transformer/transformer.py`
- `packages/ltx-core/src/ltx_core/model/transformer/transformer_args.py`

## 关键类与函数

- `LTXModel.forward()`
- `_process_transformer_blocks()`
- `_process_output()`
- `BlockPerturbationsProcessor`
- `BasicAVTransformerBlock.forward()`

## 数据流或张量流

把整条路径分成四段最容易记：

### 1. 输入段

- 调用者提交 `video: Modality | None`
- 调用者提交 `audio: Modality | None`
- 同时提交 `perturbations`

### 2. 预处理段

- `video_args_preprocessor.prepare(video, audio)`
- `audio_args_preprocessor.prepare(audio, video)`

得到两份可供 block 消费的 `TransformerArgs`。

### 3. block 堆叠段

- `_process_transformer_blocks()` 遍历 `self.transformer_blocks`
- 每层先通过 `block_input_processor` 注入本层 perturbation 状态
- 再调用 `block(video=video, audio=audio)`

### 4. 输出段

- `video_out.x` / `audio_out.x`
- 过 `_process_output()`
- 返回最终 velocity 预测 `vx, ax`

## 关键源码分段解析

### 1. `forward()` 的第一件事是做模态合法性检查

如果当前模型不支持某模态，但输入里又传了这个模态，代码会直接报错。这个检查虽然简单，却能帮你理解：LTXModel 不是默认永远双流，而是严格服从 `LTXModelType`。

### 2. 预处理器是前向主线的真正起点

从阅读视角看，`forward()` 里最重要的不是第一行 `if` 检查，而是这两句：

- `video_args_preprocessor.prepare(...)`
- `audio_args_preprocessor.prepare(...)`

因为从这一刻开始，原始输入就正式变成了后续所有 block 都会使用的标准输入格式。

### 3. `_process_transformer_blocks()` 是整条主线的中枢

这里做了几件很有代表性的工程工作：

- 当没有传 perturbations 时，自动构造空配置；
- 每进一层前，用 `block_input_processor` 预先附着本层 perturbation mask；
- 允许在训练时开启 gradient checkpointing；
- 否则就直接顺序执行 block。

所以这段函数不是“数学核心”，却是“运行组织核心”。

### 4. block 里真正发生计算，模型壳里负责调度

这一点特别值得建立心智模型：

- `LTXModel.forward()` 不负责算 attention；
- `LTXModel.forward()` 不负责算 RoPE；
- `LTXModel.forward()` 不负责决定某一层的 gate 值。

这些都下沉到 preprocessor 和 block 里。

模型壳本身的职责更像：

- 准备输入；
- 组织层级调用；
- 收集输出。

### 5. `_process_output()` 把隐藏状态变成最终预测

这一层很容易被忽略，但其实它也继续用了时间步信息。

流程是：

1. `scale_shift_table + embedded_timestep`
2. 拆成 `shift` 和 `scale`
3. 对隐藏状态做最终 `LayerNorm`
4. 再做缩放、平移和线性投影

这说明时间步调制并没有在 block 结束时就停止，而是一直延续到输出头。

## 设计动机与实现取舍

`10` 的重点是把前面拆开的机制重新接回一条完整主线，让读者可以回到 `LTXModel.forward()` 时不再迷路。

这一主线的设计很体现工程上的分层：

- `Modality` 负责输入契约；
- preprocessor 负责准备运行材料；
- block 负责层级计算；
- output head 负责生成最终预测。

这种拆分的好处是阅读和替换都更清楚。比如要换 attention backend，不需要改 `LTXModel.forward()`；要研究 cross-attention，也不需要先搞懂输出投影头。

同时，这种设计也意味着源码学习不能只盯一个文件。单看 `model.py`，你会知道“调用顺序”；但只有把它和 [01-LTXModel整体结构与调用主线.md](/Users/linkwind/Code/LTX-2/notes/ltx2-transformer/01-LTXModel整体结构与调用主线.md)、`transformer_args.py`、`transformer.py` 一起看，整条路径才会完整。

## 一句话总结与下一篇跳转

这一篇的结论是：`LTXModel.forward()` 的本质不是“亲自做所有计算”，而是把输入准备、block 堆叠和输出投影组织成一条稳定主线。下一篇会进入 [11-关键配置对象与可插拔设计.md](/Users/linkwind/Code/LTX-2/notes/ltx2-transformer/11-关键配置对象与可插拔设计.md)，看看这套主线为什么还能在不改结构的前提下切换不同算子策略。
